# Innovation Brain -- Benchmark Tests (Authoritative)

These benchmark scenarios test the Innovation Brain's ability to evaluate ideas,
design experiments, make kill/continue decisions, and manage innovation portfolios
under realistic business conditions.

Every scenario must be answered with structured analysis, not platitudes.
Responses are evaluated against the InnovationScore.md dimensions.

---

## HOW TO USE THESE BENCHMARKS

1. Present each scenario to the Innovation Brain as if it were a real business situation
2. Evaluate the response against the Expected Analysis Framework for each test
3. Score the response using InnovationScore.md dimensions
4. A passing response must address all Required Elements listed
5. Track scores over time to measure brain improvement

### Scoring Each Response
- **5/5** -- All required elements addressed, analysis rigorous, frameworks applied correctly, recommendation defensible
- **4/5** -- Most required elements addressed, analysis sound, frameworks mostly correct, minor gaps
- **3/5** -- Required elements partially addressed, analysis directional but shallow, frameworks loosely applied
- **2/5** -- Major elements missing, analysis weak, frameworks misapplied or absent
- **1/5** -- Response is generic innovation advice, no structured analysis, no frameworks

## SCENARIO 1: Idea Evaluation and Selection

**Situation:**
Innovation team completed a 2-day ideation sprint and generated 10 ideas:

1. AI-powered customer support chatbot (internal use)
2. Blockchain-based supply chain tracking for SMB
3. Mobile app for employee wellness challenges
4. API marketplace for internal microservices
5. AR-based product visualization for e-commerce
6. Automated compliance monitoring for fintech
7. Community-driven knowledge base with AI curation
8. Predictive maintenance platform for manufacturing
9. Personal finance coach using open banking data
10. B2B data enrichment API for sales teams

Company context: B2B SaaS, 200 employees, $20M ARR, project management vertical.

**Question:** Design an evaluation framework and select the top 3 ideas. Show your complete work.

### Required Elements in Response
- [ ] Evaluation criteria defined with weights (not all criteria equal)
- [ ] Strategic fit assessment (alignment with company strengths and market position)
- [ ] Market opportunity sizing for each shortlisted idea
- [ ] Feasibility assessment (technical, commercial, operational)
- [ ] Risk-adjusted ranking (not just opportunity, but opportunity vs. risk)
- [ ] Clear rationale for top 3 selection AND clear rationale for rejections
- [ ] Horizon assignment for each selected idea (H1/H2/H3)
- [ ] Next steps for each selected idea (what to test first)
- [ ] Portfolio balance assessment of the selected three

## SCENARIO 2: Kill or Continue Decision

**Situation:**
Innovation project: AI-powered contract analysis tool for legal teams.
Status: 6 months in, $200K invested (2 FTEs + cloud costs + legal dataset licensing).
Progress:
- MVP built and functional
- 8 beta users (all from personal network of founder)
- Average usage: 3 times per month per user (expected: daily)
- NPS from beta users: 35 (lukewarm)
- No paying customers yet
- One LOI from a mid-size law firm ($500/mo, not signed yet)
- 3 competitors launched similar products in the last 2 months
- Team morale declining, asking for clearer direction

Original kill criteria were: "500 beta signups in 6 months" -- missed by 98.4%.
Revised kill criteria (proposed by team): "1 paying customer in 3 months"

**Question:** Continue, pivot, or kill? Structure your analysis rigorously.

### Required Elements in Response
- [ ] Honest assessment of progress vs. expectations (no sugar-coating)
- [ ] Analysis of signal quality (8 beta users from personal network is weak evidence)
- [ ] Usage data interpretation (3x/month vs. expected daily = massive gap)
- [ ] Competitive landscape impact (3 new competitors changes the equation)
- [ ] Kill criteria analysis (original criteria missed, team proposing softer criteria = red flag)
- [ ] Sunk cost acknowledgment (do not let $200K invested influence forward decision)
- [ ] If continue: What specifically must change? What is the new kill criteria?
- [ ] If pivot: To what? Based on what signal?
- [ ] If kill: Resource reallocation recommendation
- [ ] Team morale consideration (not primary factor but must be addressed)

## SCENARIO 3: Lean Experiment Design

**Situation:**
Hypothesis: B2B buyers at mid-market companies (100-500 employees) will pay $500/month
for an AI feature that auto-generates project status reports from tool usage data.

Budget: $5,000. Timeline: 4 weeks.
You have access to: Company website (5K monthly visitors), LinkedIn (10K followers),
an email list of 2,000 existing customers, and a Slack community of 500 members.

**Question:** Design a lean experiment to validate (or invalidate) this hypothesis. Maximum rigor within constraints.

### Required Elements in Response
- [ ] Specific experiment design (not vague "talk to customers")
- [ ] Hypothesis decomposed into testable sub-assumptions
- [ ] Riskiest assumption identified and prioritized for testing
- [ ] Falsifiable success criteria with specific numbers
- [ ] Sample size justification (given constraints)
- [ ] Budget allocation across experiment activities
- [ ] Timeline with weekly milestones
- [ ] Multiple evidence types combined (not relying on single signal)
- [ ] Plan B if primary experiment fails to generate signal
- [ ] How to distinguish "bad idea" from "bad experiment"

## SCENARIO 4: Innovation Portfolio Rebalancing

**Situation:**
Current innovation portfolio (total budget: $2M/year):
- Project Alpha: H1, CRM integration suite, 12 months in, $600K invested, 50 paying customers, growing 15% MoM
- Project Beta: H2, AI analytics engine, 8 months in, $400K invested, 5 pilot users, tepid feedback
- Project Gamma: H3, blockchain credentialing platform, 3 months in, $100K invested, concept validated with 20 interviews
- Project Delta: H1, mobile app redesign, 6 months in, $300K invested, launched, adoption 30% below projections
- Project Epsilon: H2, API marketplace, 2 months in, $50K invested, market research phase

Remaining budget: $550K. Board wants to see path to $5M ARR from innovation projects within 18 months.

**Question:** Rebalance this portfolio. Show allocation decisions with rationale.

### Required Elements in Response
- [ ] Current portfolio mapped to horizons with investment amounts
- [ ] Assessment of each project (double down, maintain, reduce, kill)
- [ ] Budget reallocation with specific dollar amounts
- [ ] Horizon balance analysis (current vs. desired)
- [ ] Path to $5M ARR modeled (which projects contribute what)
- [ ] Kill decisions with clear reasoning
- [ ] Resource reallocation plan for killed projects
- [ ] Risk assessment of rebalanced portfolio
- [ ] Timeline and milestones for next 18 months

## SCENARIO 5: Assumption Mapping and Prioritization

**Situation:**
New venture idea: Subscription service that provides small restaurants with
AI-generated marketing content (social media posts, email campaigns, menu descriptions)
customized to their cuisine, location, and season. Price: $99/month.

Team is excited and wants to start building immediately.

**Question:** Map all critical assumptions, prioritize by risk, and design the first 3 experiments. Prevent premature building.

### Required Elements in Response
- [ ] Comprehensive assumption map (minimum 10 assumptions identified)
- [ ] Risk prioritization (likelihood x impact matrix)
- [ ] Top 3 riskiest assumptions identified with reasoning
- [ ] Experiment design for each of the top 3 (method, success criteria, timeline)
- [ ] Sequence recommendation (test in what order and why)
- [ ] Budget estimate for assumption testing
- [ ] Explicit argument AGAINST building until assumptions tested
- [ ] What "good enough" evidence looks like to proceed to build

## SCENARIO 6: Competitive Response to Fast Follower

**Situation:**
Your company launched an innovative feature 4 months ago: AI-powered project
timeline prediction. It was the first in the market. Early results are strong:
20% of customers adopted it, NPS for the feature is 65.

Yesterday, your largest competitor (3x your market share) announced an identical
feature launching in 30 days. Their marketing positioning claims "superior AI with
enterprise security." Your technical advantage was approximately 6 months of
proprietary training data.

**Question:** Design the innovation response strategy. How do you maintain advantage?

### Required Elements in Response
- [ ] Honest assessment of defensibility (is 6 months of data a real moat?)
- [ ] Short-term response plan (next 30 days before their launch)
- [ ] Medium-term strategy (next 6 months)
- [ ] Long-term defensibility building (network effects, data flywheel, switching costs)
- [ ] Feature roadmap acceleration vs. differentiation choice
- [ ] Customer communication strategy
- [ ] Competitive intelligence monitoring plan
- [ ] Scenario planning (what if their version is better? equivalent? worse?)
- [ ] Innovation pipeline adjustments based on this competitive reality

## SCENARIO 7: Innovation Culture Assessment

**Situation:**
CEO asks you to assess the innovation culture and propose improvements.
Current state from employee survey (200 respondents):
- "I feel safe proposing new ideas": 45% agree
- "Failed experiments are treated as learning": 30% agree
- "Innovation is rewarded": 25% agree
- "I have time for innovation work": 15% agree
- "I know how to get an idea funded": 20% agree

Additional data:
- 3 ideas submitted to innovation program in last quarter (from 200 employees)
- 0 ideas advanced past initial review
- Innovation budget exists but is unspent (team does not know how to access it)
- Last successful innovation was 2 years ago
- Leadership talks about innovation frequently but has not participated in programs

**Question:** Diagnose the root causes and design a 90-day innovation culture transformation plan.

### Required Elements in Response
- [ ] Root cause diagnosis (not just symptoms)
- [ ] Cultural barriers identified (safety, incentives, time, process)
- [ ] Leadership behavior gap identified (talk vs. walk)
- [ ] 90-day plan with weekly/monthly milestones
- [ ] Quick wins (what can change in week 1?)
- [ ] Structural changes needed (time allocation, incentive redesign, process)
- [ ] Measurement framework (how to know if culture is improving)
- [ ] Risk assessment (what could derail the transformation)
- [ ] Budget and resource requirements

## SCENARIO 8: Pivot Decision Framework

**Situation:**
Startup building an AI tutoring platform for K-12 students (B2C).
18 months in, $1.5M invested, 500 users, $8K MRR.
Problem: Customer acquisition cost is $180 (LTV is $120). Unit economics are underwater.
However: 15 schools have contacted you asking about bulk licensing for classroom use.
School interest emerged organically; you have not marketed to schools.

**Question:** Should you pivot from B2C to B2B (schools)? Design the analysis and decision framework.

### Required Elements in Response
- [ ] Current B2C assessment (unit economics, growth trajectory, path to viability)
- [ ] B2B opportunity assessment (market size, willingness to pay, sales cycle)
- [ ] Pivot cost analysis (what changes? what stays? what is lost?)
- [ ] Customer evidence comparison (500 struggling B2C users vs. 15 inbound schools)
- [ ] Business model viability comparison (B2C subscription vs. B2B licensing)
- [ ] Team capability assessment (can this team sell B2B?)
- [ ] Timeline and capital requirements for each path
- [ ] Decision framework with clear criteria (not just pros/cons)
- [ ] Hybrid option assessment (is there a path that serves both?)
- [ ] Risk analysis for each option

## SCENARIO 9: Innovation Metrics Dashboard Design

**Situation:**
VP of Innovation asks you to design the innovation metrics dashboard. The company
has 8 active innovation projects across H1/H2/H3. Current tracking is ad hoc:
some projects report monthly, others quarterly, some never. There is no
standardized way to compare projects or make portfolio decisions.

**Question:** Design the innovation metrics dashboard. What do you track, how, and why?

### Required Elements in Response
- [ ] Input metrics (what goes into innovation: budget, people, ideas)
- [ ] Process metrics (how well innovation runs: speed, experiment velocity, learning rate)
- [ ] Output metrics (what comes out: revenue, customers, validated ideas)
- [ ] Portfolio metrics (balance, allocation, pipeline health)
- [ ] Project-level metrics (per-project health indicators)
- [ ] Leading indicators (predict future outcomes)
- [ ] Lagging indicators (measure past results)
- [ ] Dashboard layout or hierarchy (what is top-level vs. drill-down)
- [ ] Cadence (what is reviewed when by whom)
- [ ] Anti-metrics (what NOT to track and why, to avoid perverse incentives)

## SCENARIO 12: Innovation Accounting

**Situation:**
Board wants ROI on the innovation program. Last year:
- Innovation budget: $3M
- Projects launched: 12
- Projects killed: 5
- Projects in market: 4
- Projects still in validation: 3
- Revenue from innovation projects: $400K ARR
- One project grew from $0 to $250K ARR in 9 months

CFO says innovation is "burning cash with minimal return."
VP Innovation says "you cannot measure innovation like a factory."
Both have valid points but neither is providing useful analysis.

**Question:** Build an innovation accounting framework that satisfies both perspectives.

### Required Elements in Response
- [ ] Framework that acknowledges innovation's non-linear returns
- [ ] Metrics the CFO can track (financial, not just activity)
- [ ] Metrics the VP Innovation can defend (learning, not just revenue)
- [ ] Portfolio-level return analysis (not just per-project)
- [ ] Time-adjusted analysis (innovation takes time, measure appropriately)
- [ ] Comparison to alternative investments (what if $3M was spent on core product?)
- [ ] Option value calculation (what is the value of validated bets, not just revenue?)
- [ ] Kill rate analysis (is 5/12 kill rate healthy? benchmark it)
- [ ] Recommendation for next year's budget with rationale

## SCENARIO 13: Customer Co-Creation Program

**Situation:**
Product team wants to involve customers in innovation. Currently, customer input
comes from: support tickets (reactive), NPS surveys (vague), and annual customer
advisory board (12 customers, meets twice a year, dominated by largest customers).

VP Product wants a "customer co-creation program" but is worried about:
- Customers designing by committee
- Vocal minority driving roadmap
- IP concerns with sharing early concepts
- Managing expectations for features that get killed

**Question:** Design a customer co-creation program that addresses all concerns.

### Required Elements in Response
- [ ] Program structure (who participates, how selected, commitments)
- [ ] Guard against design-by-committee (structured input, not open voting)
- [ ] Representativeness strategy (diverse segments, not just loudest)
- [ ] IP protection framework (NDAs, what is shared, what is not)
- [ ] Expectation management (how to share concepts without promising delivery)
- [ ] Integration with innovation process (when does customer input enter?)
- [ ] Incentive structure for participating customers
- [ ] Measurement framework (is the program improving innovation quality?)
- [ ] Scaling approach (start small, expand based on results)
- [ ] Kill communication (how to tell co-creation customers a feature was killed)

## SCENARIO 14: Moonshot vs. Incremental Balance

**Situation:**
Engineering team produces a steady stream of incremental improvements (H1).
Leadership complains there is "no big vision." When leadership proposes moonshots,
engineering says they are "unrealistic" and "not grounded in customer reality."

Last 3 moonshot attempts:
- Autonomous AI agent (abandoned after 4 months, technically infeasible at the time)
- Blockchain integration (abandoned after 2 months, no customer demand)
- New market entry (abandoned after 6 months, could not find product-market fit)

Engineering has a point: moonshots keep failing. Leadership has a point: incremental
improvements will not 10x the business.

**Question:** Design a framework that bridges the gap. How do you do responsible moonshots?

### Required Elements in Response
- [ ] Root cause analysis of why previous moonshots failed
- [ ] Framework for "responsible moonshots" (high ambition, structured validation)
- [ ] Budget and resource allocation model (ring-fence moonshot investment)
- [ ] Stage-gate process for moonshots (different from incremental gates)
- [ ] Team structure for moonshots (separate from core? hybrid? rotation?)
- [ ] How to connect moonshots to customer reality (not pure speculation)
- [ ] Kill criteria specific to moonshots (faster, more disciplined)
- [ ] How to celebrate moonshot learning even when the project dies
- [ ] Metrics for moonshot program health (not just success/failure)
- [ ] Bridge between engineering pragmatism and leadership ambition

## SCENARIO 15: Post-Mortem Excellence

**Situation:**
Major innovation project just got killed after 12 months and $800K investment.
The project (AI-powered workforce planning tool) had strong initial customer interest
but failed to achieve product-market fit. Team of 4 is demoralized. Other innovation
teams are watching to see how this is handled.

**Question:** Design and facilitate the post-mortem. Extract maximum learning. Maintain innovation culture.

### Required Elements in Response
- [ ] Post-mortem structure (timeline, attendees, format)
- [ ] Blameless analysis framework (focus on system, not individuals)
- [ ] Key questions to answer (what did we learn? what signals did we miss?)
- [ ] Learning extraction template (codified lessons, not just discussion notes)
- [ ] Pattern identification (does this failure connect to other failures?)
- [ ] Process improvement recommendations (what changes prevent similar failures?)
- [ ] Team treatment plan (redeployment, recognition, decompression)
- [ ] Communication to broader organization (celebrate learning, not just success)
- [ ] Documentation for institutional memory
- [ ] Specific lessons to add to the Innovation Brain's evaluation criteria
