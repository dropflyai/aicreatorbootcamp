# EVIDENCE PROTOCOL
**Taxonomy, Collection, Storage, and Sufficiency of Verification Artifacts**

---

## 1. Purpose

Evidence is the material record that a claimed behavior was observed, a
condition was tested, and a result was obtained. In the Engineering Brain,
claims without evidence are prohibited (Constitution.md Section 9). This
document defines what constitutes valid evidence, how to collect it, where
to store it, when it is sufficient, and how long to retain it.

The evidence protocol exists because:

- Memory is unreliable; artifacts are not.
- Verbal assurances ("it works") cannot be audited.
- Regression investigation requires historical evidence to compare against.
- Compliance and security audits require demonstrable proof of diligence.

Evidence is the difference between engineering and guesswork.

## 2. Evidence Types Taxonomy

### 2.1 Test Reports

**Format**: JUnit XML, TAP (Test Anything Protocol), JSON test output.

**Contents**: Test name, status (pass/fail/skip/error), duration, failure
message with stack trace, timestamp, environment metadata.

**Generation**:
- pytest: `pytest --junitxml=reports/test-results.xml`
- jest: `jest --reporters=jest-junit` (with `jest-junit` package)
- vitest: `vitest run --reporter=junit --outputFile=reports/test-results.xml`

**Validity criteria**:
- Timestamp within the current CI run or development session.
- Test count matches or exceeds the expected count (no silently dropped tests).
- Zero failures for Layer 1 pass.

### 2.2 Coverage Reports

**Format**: Cobertura XML, LCOV, Istanbul JSON, HTML summary.

**Contents**: Line coverage, branch coverage, function coverage per file.
Aggregate totals. Uncovered line ranges.

**Generation**:
- pytest: `pytest --cov=src --cov-report=xml:reports/coverage.xml`
- istanbul/nyc: `nyc report --reporter=cobertura`
- vitest: `vitest run --coverage --coverage.reporter=cobertura`

**Validity criteria**:
- Coverage percentage meets or exceeds the project threshold (default 80%).
- No regression from the previous measurement (coverage must not decrease
  without documented justification).

### 2.3 Screenshots

**Format**: PNG (lossless). JPEG is acceptable only when file size is a
binding constraint.

**Contents**: Visual capture of UI state. Must include the complete viewport
or the relevant component in isolation.

**Subtypes**:
- **Before/After**: Captured before and after a change to demonstrate
  visual difference. Required for all UI layout changes.
- **Baseline**: The canonical correct rendering, stored as the reference for
  visual regression testing.
- **Failure diff**: The computed difference between baseline and actual,
  generated by visual regression tools.

**Generation**:
- Playwright: `await page.screenshot({ path: 'evidence/screenshot.png', fullPage: true })`
- Chromium DevTools Protocol: headless capture via CDP.
- Manual capture: Only acceptable under GEAR: EXPLORE.

**Validity criteria**:
- Resolution matches the target viewport (e.g., 1280x720, 375x812).
- No browser chrome or OS artifacts unless explicitly relevant.
- Timestamp in EXIF or filename.

### 2.4 Performance Benchmarks

**Format**: JSON, CSV, or structured text. Must include numerical measurements
with units.

**Contents**: Metric name, measured value, unit, baseline value, delta,
percentage change, pass/fail against threshold.

**Metric categories**:
- **Latency**: API response time (p50, p95, p99), page load time, Time to
  First Byte (TTFB), Largest Contentful Paint (LCP).
- **Throughput**: Requests per second, operations per second.
- **Resource**: Memory usage (RSS, heap), CPU utilization, bundle size.
- **Database**: Query execution time, connection pool utilization.

**Generation**:
- k6: `k6 run --out json=reports/benchmark.json load-test.js`
- Lighthouse: `lighthouse --output=json --output-path=reports/lighthouse.json`
- Custom scripts: Must output structured data, not prose.

**Validity criteria**:
- Baseline comparison is present (absolute numbers without context are
  insufficient).
- Environment is documented (hardware, OS, network conditions).
- Statistical significance: minimum 3 runs for latency measurements.
- Regression threshold: performance must not degrade beyond the defined
  budget (default 10% regression tolerance).

### 2.5 Security Scan Results

**Format**: SARIF (Static Analysis Results Interchange Format), JSON, or
tool-specific structured output.

**Subtypes**:
- **SAST (Static Application Security Testing)**: Source code analysis for
  vulnerabilities. Tools: Bandit (Python), Semgrep, ESLint security plugins.
- **DAST (Dynamic Application Security Testing)**: Runtime analysis against
  a running application. Tools: OWASP ZAP, Nuclei.
- **SCA (Software Composition Analysis)**: Dependency vulnerability scanning.
  Tools: npm audit, pip-audit, Trivy, Snyk.
- **Secrets detection**: Scanning for leaked credentials. Tools: TruffleHog,
  detect-secrets, gitleaks.

**Generation**:
- Bandit: `bandit -r src/ -f json -o reports/sast.json`
- npm audit: `npm audit --json > reports/sca.json`
- Trivy: `trivy fs --format json --output reports/trivy.json .`
- gitleaks: `gitleaks detect --report-format json --report-path reports/secrets.json`

**Validity criteria**:
- Zero high or critical findings for Layer 1 pass.
- Medium findings documented with risk acceptance or remediation plan.
- Scan covers the full scope of changed files and dependencies.

### 2.6 Accessibility Audit Results

**Format**: JSON (axe-core output), HTML report.

**Contents**: Violation type, WCAG criterion, severity (critical, serious,
moderate, minor), affected element selector, remediation guidance.

**Generation**:
- axe-core via Playwright: See UIAutomation.md Section 5.
- Lighthouse accessibility audit: `lighthouse --only-categories=accessibility`
- pa11y: `pa11y --reporter=json <url>`

**Validity criteria**:
- Zero critical or serious violations for WCAG 2.1 AA compliance.
- Moderate violations documented with remediation timeline.
- Audit covers all pages/views affected by the change.

### 2.7 Migration Artifacts

**Format**: SQL diff, migration log output, schema comparison report.

**Contents**: Migration file(s) applied, before/after schema state,
execution output, rollback verification.

**Generation**:
- Supabase: `supabase db diff`, `supabase migration up` output
- Prisma: `prisma migrate dev` output, `prisma db pull` schema diff
- Raw SQL: Execution log with row counts and timing

**Validity criteria**:
- Migration applied successfully (exit code 0).
- Schema matches expected state after migration.
- Rollback was tested or a rollback script exists.

## 3. Collection Automation

### 3.1 CI Artifact Collection

Evidence collection must be automated in CI pipelines. Manual evidence
generation is a governance violation under GEAR: SHIP.

```yaml
# Example: GitHub Actions artifact collection
- name: Upload test reports
  uses: actions/upload-artifact@v4
  if: always()
  with:
    name: test-reports
    path: reports/
    retention-days: 30

- name: Upload screenshots
  uses: actions/upload-artifact@v4
  if: always()
  with:
    name: screenshots
    path: evidence/screenshots/
    retention-days: 90
```

### 3.2 Playwright Screenshot Automation

```typescript
// Automated evidence collection in Playwright tests
import { test, expect } from '@playwright/test';

test('capture evidence for feature X', async ({ page }) => {
  await page.goto('/feature-x');

  // Before state (if applicable)
  await page.screenshot({
    path: 'evidence/screenshots/feature-x-before.png',
    fullPage: true,
  });

  // Perform action
  await page.click('[data-testid="submit-button"]');

  // After state
  await page.screenshot({
    path: 'evidence/screenshots/feature-x-after.png',
    fullPage: true,
  });

  // Accessibility evidence
  // (see UIAutomation.md for axe-core integration)
});
```

### 3.3 Benchmark Automation

Performance benchmarks must be collected in reproducible environments.
Local benchmarks are acceptable for GEAR: BUILD. CI benchmarks are required
for GEAR: SHIP.

## 4. Storage Conventions

### 4.1 Directory Structure

```
project-root/
  evidence/
    screenshots/
      {feature-name}-{viewport}-{timestamp}.png
      {feature-name}-{viewport}-baseline.png
      {feature-name}-{viewport}-diff.png
    reports/
      test-results-{timestamp}.xml
      coverage-{timestamp}.xml
      sast-{timestamp}.json
      sca-{timestamp}.json
      accessibility-{timestamp}.json
    benchmarks/
      {benchmark-name}-{timestamp}.json
      {benchmark-name}-baseline.json
    migrations/
      {migration-id}-output.log
      {migration-id}-schema-diff.sql
```

### 4.2 Naming Conventions

- Use kebab-case for all evidence filenames.
- Include ISO 8601 timestamps: `YYYY-MM-DDTHH-MM-SS` (colons replaced with
  hyphens for filesystem compatibility).
- Include the commit SHA (short form, 7 characters) when evidence is tied to
  a specific commit: `feature-x-a1b2c3d-2026-02-19T14-30-00.png`.
- Baselines omit timestamps (they are the canonical reference).

### 4.3 Gitignore Policy

Evidence directories should be gitignored in most cases. Evidence is ephemeral
and tied to CI runs.

Exceptions:
- **Baseline screenshots** for visual regression testing: committed to the
  repository.
- **Benchmark baselines**: committed to enable regression detection.

```gitignore
# .gitignore
evidence/screenshots/*.png
!evidence/screenshots/*-baseline.png
evidence/reports/
evidence/benchmarks/*.json
!evidence/benchmarks/*-baseline.json
evidence/migrations/
```

## 5. Sufficiency Matrix

The Sufficiency Matrix defines the minimum evidence required for each change
type to satisfy Layer 2 of Triple Verification.

| Change Type | Test Report | Coverage | Screenshot | Benchmark | Security Scan | Accessibility | Migration Log |
|-------------|:-----------:|:--------:|:----------:|:---------:|:-------------:|:-------------:|:-------------:|
| Backend logic | REQUIRED | REQUIRED | -- | -- | REQUIRED | -- | -- |
| API endpoint | REQUIRED | REQUIRED | -- | RECOMMENDED | REQUIRED | -- | -- |
| UI component | REQUIRED | RECOMMENDED | REQUIRED | -- | -- | REQUIRED | -- |
| UI layout | REQUIRED | -- | REQUIRED | -- | -- | REQUIRED | -- |
| Database migration | REQUIRED | -- | -- | -- | -- | -- | REQUIRED |
| Performance change | REQUIRED | -- | -- | REQUIRED | -- | -- | -- |
| Security fix | REQUIRED | REQUIRED | -- | -- | REQUIRED | -- | -- |
| Dependency update | REQUIRED | -- | -- | -- | REQUIRED | -- | -- |
| Config change | REQUIRED | -- | -- | -- | RECOMMENDED | -- | -- |
| Infrastructure | -- | -- | -- | -- | REQUIRED | -- | -- |
| Documentation only | -- | -- | -- | -- | -- | -- | -- |

Legend:
- **REQUIRED**: Must be present and valid. Absence fails Layer 2.
- **RECOMMENDED**: Should be present. Absence is noted but does not fail Layer 2.
- **--**: Not applicable for this change type.

## 6. Evidence Chain of Custody

### 6.1 Concept

Evidence chain of custody ensures that artifacts are traceable from generation
to consumption. An artifact without provenance is inadmissible.

### 6.2 Provenance Requirements

Every evidence artifact must have:

1. **Generator**: The tool or process that created it (e.g., "pytest 8.1.0",
   "Playwright 1.42.0", "Bandit 1.7.8").
2. **Timestamp**: When it was generated (ISO 8601).
3. **Source commit**: The git commit SHA the artifact corresponds to.
4. **Environment**: OS, runtime version, CI runner ID (where applicable).
5. **Integrity**: SHA-256 hash of the artifact (for tamper detection in
   compliance-sensitive contexts).

### 6.3 Provenance Recording

In CI, provenance is recorded via the pipeline metadata (job ID, run ID,
commit SHA). Locally, provenance is recorded in a manifest file:

```json
{
  "artifact": "evidence/screenshots/dashboard-1280x720-a1b2c3d.png",
  "generator": "playwright@1.42.0",
  "timestamp": "2026-02-19T14:30:00Z",
  "commit": "a1b2c3d",
  "environment": {
    "os": "Ubuntu 22.04",
    "node": "20.11.0",
    "browser": "chromium@125.0.6422.26"
  },
  "sha256": "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
}
```

### 6.4 Tampering Detection

If an artifact's SHA-256 hash does not match the recorded hash in the
manifest, the artifact is considered compromised and must be regenerated.
This is primarily relevant in regulated or compliance-critical environments.

## 7. Retention Policy

### 7.1 Retention Periods

| Artifact Category | Retention Period | Rationale |
|-------------------|-----------------|-----------|
| Test reports | 30 days | Short-term debugging and regression investigation |
| Coverage reports | 30 days | Trend analysis over recent development |
| Screenshots (non-baseline) | 90 days | Visual regression investigation |
| Screenshots (baseline) | Indefinite (committed) | Reference for visual regression |
| Benchmarks (non-baseline) | 90 days | Performance trend analysis |
| Benchmarks (baseline) | Indefinite (committed) | Reference for performance regression |
| Security scans | 180 days | Compliance audit trail |
| Accessibility audits | 180 days | Compliance audit trail |
| Migration logs | 365 days | Schema history and rollback reference |

### 7.2 Cleanup Automation

CI pipelines should enforce retention via artifact expiration settings.
Local evidence directories should be cleaned periodically:

```bash
# Remove evidence older than retention period
find evidence/reports/ -mtime +30 -delete
find evidence/screenshots/ -mtime +90 ! -name '*-baseline*' -delete
find evidence/benchmarks/ -mtime +90 ! -name '*-baseline*' -delete
```

### 7.3 Archival

For compliance-sensitive projects, artifacts exceeding retention may be
archived to cold storage rather than deleted. Archival policy is project-
specific and defined in the project's governance documentation.

## 8. Evidence Absence Protocol

When required evidence cannot be produced:

1. **Determine why**: Is the tooling broken? Is the system untestable?
   Is the environment unavailable?
2. **Attempt remediation**: Fix the tooling, adjust the test, provision the
   environment.
3. **If remediation fails**:
   - Under GEAR: SHIP: This is a **stop condition**. Work cannot proceed.
   - Under GEAR: BUILD: File a justified waiver per Constitution.md Section 14.
     Document what evidence is missing, why, and when it will be provided.
   - Under GEAR: EXPLORE: Note the absence. No formal waiver required.
   - Under GEAR: HOTFIX: Defer evidence collection to post-incident review.
     Track in `Memory/RegressionHistory.md`.

Evidence absence is always a signal. It may indicate undertestable code,
broken automation, or insufficient infrastructure. Each instance should
be investigated for systemic causes.
