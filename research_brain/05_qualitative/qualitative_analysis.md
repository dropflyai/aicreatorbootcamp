# Qualitative Analysis

Systematic frameworks for analyzing qualitative data, from thematic coding through synthesis and presentation of findings.

---

## 1. Thematic Analysis (Braun & Clarke)

### 1.1 Overview

Thematic analysis is the most widely used qualitative analysis method in applied research. Braun and Clarke's (2006) six-phase approach provides a rigorous, systematic framework that is accessible to researchers across experience levels. It is method-agnostic (can be used with interview, focus group, observational, or textual data).

### 1.2 Phase 1: Familiarization with the Data

Before coding, immerse yourself in the data. Read and re-read transcripts. Listen to recordings. Review field notes. Active reading means making notes about initial ideas, patterns, and surprises.

**Practices:**
- Read each transcript at least twice before coding begins.
- Make margin notes on first impressions, questions, and connections.
- Write a brief memo for each participant/data source summarizing key points.
- Note emotional responses to the data (reflexivity).
- Do not begin coding prematurely. Familiarization prevents superficial analysis.

### 1.3 Phase 2: Generating Initial Codes

Systematically code the entire dataset. A code is a concise label that captures a meaningful segment of data relevant to the research questions.

**Coding approaches:**
- **Inductive coding:** Codes emerge from the data. No pre-defined codebook. The researcher generates codes as they encounter meaningful content. Best for exploratory research.
- **Deductive coding:** Codes are derived from a pre-existing framework, theory, or research question. Applied to the data to test or extend the framework. Best when building on prior research.
- **Hybrid coding:** Start with a deductive framework, then add inductive codes for data that does not fit. Most practical for applied research.

**Coding best practices:**
- Code everything relevant, even if you are not sure it will matter.
- Code for as many potential themes as possible. It is easier to collapse codes than to recode the entire dataset.
- Preserve context. A code should make sense when removed from the surrounding text.
- Code inclusively. If in doubt, include the segment.
- Maintain a codebook: code name, definition, example, decision rules.

**Granularity:** Codes should be specific enough to be meaningful but general enough to apply across multiple data sources. "Frustrated with onboarding" is better than "frustrated" (too broad) or "frustrated when the setup wizard crashed on step 3 on Tuesday" (too specific).

### 1.4 Phase 3: Searching for Themes

Group codes into candidate themes. A theme captures a coherent, recurring pattern of meaning across the dataset. Themes are not codes; they are higher-order constructs that organize and interpret coded data.

**Process:**
1. Sort all codes onto a visual workspace (physical wall, Miro, FigJam).
2. Group related codes together.
3. Identify candidate themes by examining the clusters.
4. Create a thematic map showing relationships between themes and sub-themes.
5. Collect all data extracts relevant to each candidate theme.

**Theme vs code distinction:**
- Code: "Feels overwhelmed by feature count." Another code: "Cannot find the feature I need."
- Theme: "Feature discoverability as a barrier to adoption."

### 1.5 Phase 4: Reviewing Themes

Test the candidate themes against the data. This is quality control.

**Level 1 review:** Re-read all data extracts collated under each theme. Do they form a coherent pattern? If not, reconsider: Is this one theme or two? Should some extracts be reassigned? Is there enough evidence to support this theme?

**Level 2 review:** Re-read the entire dataset with the thematic map in mind. Does the map accurately represent the data? Are there data segments that were not captured by any theme? Modify, merge, split, or discard themes as needed.

**Decision criteria:**
- A theme must be supported by data from multiple participants (not just one outlier).
- A theme must be relevant to the research questions.
- Themes should be distinct from each other (minimal overlap).
- The thematic map should tell a coherent story about the data.

### 1.6 Phase 5: Defining and Naming Themes

Write a detailed description of each theme. Define its scope, meaning, and boundaries. Identify the narrative each theme tells about the data.

**Theme definition template:**
- **Name:** Concise, descriptive (not a code label or a full sentence).
- **Definition:** What this theme captures and what it does not capture.
- **Scope:** What data falls within this theme.
- **Story:** The narrative this theme tells about the phenomenon.
- **Relationship:** How this theme relates to other themes and to the overall research question.
- **Key data extracts:** 3-5 exemplary quotes or observations that represent the theme.

### 1.7 Phase 6: Producing the Report

Write the analysis as a coherent narrative, not a list of themes. Themes should be woven into a story that answers the research questions.

**Report structure per theme:**
1. Introduce the theme and its significance.
2. Present the argument with evidence (data extracts with interpretation).
3. Connect to other themes and to the broader research context.
4. Draw out implications.

**Evidence standards:**
- Every claim must be supported by at least one data extract (direct quote or observation).
- Select vivid, representative extracts, not cherry-picked extremes.
- Present extracts with enough context to understand them.
- Analytic commentary should accompany extracts (do not let data "speak for itself").

## 2. Affinity Diagramming

### 2.1 Overview

Affinity diagramming (KJ Method, developed by Jiro Kawakita) organizes large amounts of qualitative data into natural groupings through collaborative sorting. Less rigorous than formal thematic analysis but faster and excellent for team-based synthesis workshops.

### 2.2 Process

1. Write each observation, insight, or data point on a separate sticky note (physical or digital).
2. All team members silently sort notes into groups based on natural affinity (no talking during initial sort to prevent dominant voices from influencing groupings).
3. Once groupings stabilize, discuss and adjust as a team.
4. Name each group with a descriptive header.
5. Identify relationships between groups (hierarchies, dependencies, tensions).
6. Prioritize groups by frequency, impact, or strategic relevance.

**Best for:** Workshop-based synthesis, design sprints, collaborative sense-making, translating research into design opportunities.

## 3. Constant Comparative Method

### 3.1 Origin and Application

Developed as part of grounded theory (Glaser & Strauss, 1967), the constant comparative method systematically compares new data instances to existing codes and categories throughout the analysis process.

### 3.2 Comparison Levels

**Comparing instances within a code:** Does this new instance fit the existing code definition? If not, should the code definition be expanded or should a new code be created?

**Comparing codes to codes:** How are these codes related? Do they represent the same underlying concept? Should they be merged? Are they dimensions of a higher-order category?

**Comparing categories to categories:** How do categories relate to each other? What is the central phenomenon? How do categories connect in a theoretical model?

**Comparing data to the emerging theory:** Does the new data support, refine, or challenge the emerging explanation? If it challenges, how should the theory be modified?

## 4. Coding in Practice

### 4.1 Open Coding

The initial pass through data, generating codes without pre-determined structure. Code every meaningful segment. Over-code rather than under-code. The goal is to break the data open.

**Techniques:**
- Line-by-line coding: Code every line. Extremely thorough but time-intensive. Best for the first few transcripts.
- Paragraph coding: Code each paragraph or thought unit. More efficient for large datasets after initial familiarity.
- In vivo coding: Use the participant's own words as codes. Preserves authentic voice and reveals indigenous concepts.

### 4.2 Axial Coding

Second pass: identify relationships between codes. Organize codes into categories and sub-categories. Map the properties and dimensions of each category. Explore conditions, actions/interactions, and consequences.

### 4.3 Selective Coding

Third pass: identify the core category that integrates all others. Develop the theoretical story. Determine which categories are central and which are peripheral. Refine until the theory accounts for variation in the data.

### 4.4 Codebook Management

Maintain a living codebook throughout analysis:

| Code | Definition | When to Use | When NOT to Use | Example |
|------|-----------|-------------|-----------------|---------|
| TRUST-EROSION | Participant describes declining trust in the product or company | When trust decrease is attributed to a specific experience | General dissatisfaction without trust element | "After that data breach, I just don't believe them anymore" |

## 5. Qualitative Data Management Tools

### 5.1 Dedicated QDAS (Qualitative Data Analysis Software)

**NVivo:** Industry standard for academic and applied qualitative research. Supports coding, querying, visualization, team collaboration. Steep learning curve. Best for large datasets and complex analyses.

**Atlas.ti:** Strong visualization and network features. Good for theory building and relationship mapping. Available on desktop and cloud.

**Dovetail:** Purpose-built for UX and product research. Combines repository, analysis, and insight management. Best for applied research teams integrating research into product development.

**MAXQDA:** Strong mixed-methods support. Good for researchers who combine qualitative and quantitative analysis.

### 5.2 Lightweight Alternatives

**Spreadsheets:** Simple but functional. Columns for data, codes, themes, memos. Works for small datasets (<20 transcripts).

**Miro/FigJam:** Visual collaboration tools. Excellent for affinity diagramming and collaborative coding. Not designed for systematic coding but effective for team workshops.

**Notion/Airtable:** Database-based approaches. Tags function as codes. Better for repository management than deep analysis.

## 6. Intercoder Reliability

### 6.1 Purpose

When multiple researchers code the same data, intercoder reliability measures the degree of agreement. It ensures that coding reflects the data rather than individual researcher bias.

### 6.2 Process

1. Develop a codebook with clear definitions and examples.
2. Two or more coders independently code the same subset of data (typically 10-20% of the dataset).
3. Calculate agreement using Cohen's kappa (two coders) or Fleiss' kappa (three or more).
4. Discuss disagreements to refine codebook definitions.
5. Recode after refinement. Repeat until acceptable reliability is achieved.

### 6.3 Benchmarks

| Kappa Value | Interpretation |
|------------|---------------|
| < 0.20 | Poor agreement |
| 0.21-0.40 | Fair agreement |
| 0.41-0.60 | Moderate agreement |
| 0.61-0.80 | Substantial agreement |
| 0.81-1.00 | Almost perfect agreement |

**Target:** Kappa > 0.70 for applied research. For high-stakes research, target > 0.80.

## 7. Member Checking

### 7.1 Purpose and Process

Member checking involves sharing findings with participants to verify that the researcher's interpretation accurately represents their experience. It is a key strategy for establishing credibility (internal validity).

### 7.2 Approaches

**Transcript review:** Share the transcript with the participant for accuracy verification. Participants can correct factual errors or clarify ambiguous statements.

**Interpretive member checking:** Share themes or a summary of findings with participants. Ask: "Does this ring true to your experience?" This is more valuable than transcript review because it validates interpretation, not just transcription.

**Participatory analysis:** Involve participants in the analysis process. Co-create themes and interpretations. Used in participatory action research.

## 8. Saturation

### 8.1 Definition

Theoretical saturation occurs when new data no longer generates new codes, themes, or theoretical insights. It is the primary criterion for determining adequate sample size in qualitative research.

### 8.2 Practical Assessment

- Track the rate of new code generation across interviews.
- Plot a saturation curve: cumulative new codes vs interview number.
- When the curve plateaus (2-3 consecutive interviews with no new codes), saturation is approaching.
- Guest, Bunce, and Johnson (2006) found that 12 interviews typically achieve saturation in a relatively homogeneous population. More heterogeneous populations require more.

### 8.3 Saturation Is Not an Excuse for Under-Sampling

Claiming saturation after 5 interviews is rarely credible. Plan for a minimum of 12-15 per segment and assess saturation empirically rather than declaring it prematurely.

## 9. Presenting Qualitative Findings

### 9.1 Report Structure

1. **Research questions and methodology** (brief).
2. **Participant summary** (demographics, recruitment, without identifying details).
3. **Thematic overview** (map or table of themes).
4. **Detailed theme analysis** (narrative with embedded evidence).
5. **Discussion** (interpretation, connection to prior research, theoretical implications).
6. **Recommendations** (actionable implications for stakeholders).
7. **Limitations and future research.**

### 9.2 Presenting to Business Stakeholders

Executives and product teams need qualitative findings distilled into clear, actionable insights:

- Lead with the "so what" -- what should the business do differently.
- Use 3-5 key findings, each supported by 1-2 powerful quotes.
- Create a video highlight reel (2-3 minutes) of participant moments.
- Present as a journey map, empathy map, or persona when appropriate.
- Avoid academic language. "We found" instead of "The data revealed through inductive thematic analysis."

## 10. Qualitative Analysis Quality Checklist

- [ ] Analysis method is explicitly stated and appropriate for the research question
- [ ] Coding process is documented (codebook maintained throughout)
- [ ] Intercoder reliability is assessed (kappa reported)
- [ ] Themes are reviewed against the data (Phase 4 of Braun & Clarke)
- [ ] Member checking or other credibility strategies are employed
- [ ] Saturation is assessed and documented
- [ ] Findings are grounded in data (every claim has supporting evidence)
- [ ] Analytic commentary accompanies data extracts (not just listing quotes)
- [ ] Reflexivity is documented
- [ ] Limitations are acknowledged honestly
