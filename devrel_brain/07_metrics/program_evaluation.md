# Program Evaluation

## Overview

Program evaluation is the systematic assessment of DevRel programs, initiatives, and experiments to determine their effectiveness, efficiency, and business impact. Unlike individual metric tracking, program evaluation examines entire programs holistically: Did this hackathon program achieve its goals? Is our documentation investment paying off? Should we continue our conference sponsorship strategy? This module covers evaluation frameworks, ROI methodology, experiment design, program lifecycle management, and the organizational decision-making processes that determine which programs live, grow, or die.

---

## 1. Evaluation Frameworks

### 1.1 The Kirkpatrick Model for DevRel

Adapted from the training evaluation model (Kirkpatrick, 1959), applied to DevRel programs:

| Level | Question | Evaluation Method | Timeline |
|-------|---------|-------------------|----------|
| **Level 1: Reaction** | Did developers enjoy the experience? | Post-event survey, NPS, social sentiment | Immediate |
| **Level 2: Learning** | Did developers learn what we intended? | Quiz, tutorial completion, skill assessment | Days |
| **Level 3: Behavior** | Did developers change their behavior? | Activation rate, API usage, feature adoption | Weeks-Months |
| **Level 4: Results** | Did the program achieve business outcomes? | Revenue, retention, growth metrics | Months-Quarters |

**Key Insight:** Most DevRel programs measure only Level 1 (did they enjoy it). Mature programs measure through Level 4 (did it drive business results). The jump from Level 1 to Level 4 is where most DevRel teams fail.

### 1.2 The Logic Model

A logic model connects program inputs to outcomes:

```
INPUTS          ACTIVITIES        OUTPUTS          OUTCOMES          IMPACT
(What we        (What we          (What we         (What changed)    (Business
 invest)         do)               produce)                          result)

$50K budget     Write 12          12 blog          15K page views    500 new
2 advocates     blog posts        posts            2K signups        activated
100 hours       Host 4            published         300 activations   developers
                workshops         4 workshops                        $75K
                Build 2           held              200 attendees     attributed
                integrations      2 integrations    got activated     revenue
                                  launched
```

### 1.3 Program Maturity Assessment

| Dimension | Level 1: Starting | Level 2: Operating | Level 3: Optimizing |
|-----------|-------------------|-------------------|---------------------|
| Strategy | Ad hoc activities | Documented strategy | Data-driven strategy |
| Metrics | No measurement | Basic tracking (views, likes) | Full funnel with attribution |
| Processes | Reactive | Defined playbooks | Continuous improvement |
| Team | 1 person wearing all hats | Specialized roles | Cross-functional integration |
| Budget | Zero or informal | Dedicated budget | ROI-justified budget |

## 2. ROI Methodology

### 2.1 DevRel ROI Formula

```
DevRel ROI = (Value Generated by DevRel - DevRel Investment) / DevRel Investment x 100

Where:
  Value Generated = Direct Revenue + Pipeline Influence + Cost Savings + Brand Value
  DevRel Investment = People + Tools + Events + Content + Travel + Overhead
```

### 2.2 Value Components

**Direct Revenue Attribution:**
```
Revenue from developers acquired through DevRel channels:
  = Σ (developer_revenue × attribution_weight)

Example:
  1,000 developers signed up through blog content (first-touch)
  300 activated (30% activation rate)
  45 converted to paid (15% conversion)
  Average annual revenue per paid developer: $2,400
  Total direct revenue: 45 × $2,400 = $108,000
  Attribution weight (first-touch): 0.4
  DevRel-attributed revenue: $108,000 × 0.4 = $43,200
```

**Pipeline Influence:**
```
Enterprise deals where a developer championed the product internally:
  = Σ (deal_value × influence_weight)

Example:
  10 enterprise deals closed this quarter
  6 had a developer champion who came through DevRel
  Average deal size: $50,000
  Influence weight: 0.3 (DevRel was one of many factors)
  Pipeline influence: 6 × $50,000 × 0.3 = $90,000
```

**Cost Savings:**
```
Support tickets avoided through documentation and community:
  = (tickets_avoided × cost_per_ticket)

Example:
  Community answered 500 questions that would have been tickets
  Average ticket cost: $15
  Cost savings: 500 × $15 = $7,500/month = $90,000/year
```

**Brand Value:**
```
Equivalent advertising cost for organic reach:
  = (total_impressions × equivalent_CPM / 1000)

Example:
  Blog: 500K impressions × $10 CPM = $5,000
  Social: 2M impressions × $5 CPM = $10,000
  YouTube: 100K views × $20 CPV = $2,000 (estimated)
  Total brand value equivalent: $17,000/month
```

### 2.3 Investment Components

| Component | Calculation | Example |
|-----------|------------|---------|
| People | FTE count × fully loaded cost | 3 FTEs × $150K = $450K |
| Events | Sponsorships + travel + production | $100K |
| Tools | Platform subscriptions and licenses | $30K |
| Content | Production costs (video, design, guest writers) | $40K |
| Travel | Non-event travel (meetups, partner visits) | $25K |
| Overhead | Office, equipment, management allocation | $55K |
| **Total** | | **$700K** |

### 2.4 ROI Calculation Example

```
Total Value Generated:
  Direct revenue: $43,200
  Pipeline influence: $90,000
  Cost savings: $90,000
  Brand value: $204,000 (annualized)
  Total: $427,200

Total Investment: $700,000

ROI = ($427,200 - $700,000) / $700,000 × 100 = -39%

Wait -- is this a failure? Not necessarily. Consider:
  1. DevRel has a 6-18 month lag. Year 1 ROI is almost always negative.
  2. The developers acquired in Year 1 generate revenue in Years 2-5.
  3. LTV of a developer acquired through DevRel is typically 2-3x higher
     than paid acquisition (because they are educated and self-serve).

Adjusted Year 1 ROI (with 3-year LTV projection):
  Year 1 revenue: $43,200
  Year 2 projected: $108,000 (compound growth)
  Year 3 projected: $162,000
  3-year LTV: $313,200
  Pipeline + savings + brand (3-year): $1,152,000
  3-year total value: $1,465,200
  3-year investment: $2,100,000 (assuming flat)
  3-year ROI: -30% to positive depending on growth rate
```

## 3. Program-Level Evaluation

### 3.1 Content Program Evaluation

| Metric | How to Measure | Evaluation Criteria |
|--------|---------------|-------------------|
| Content volume | Posts/videos published per month | Are we hitting our publishing cadence? |
| Content quality | Time on page, completion rate, NPS | Is the content good enough? |
| Content reach | Unique visitors, social shares | Is the content finding its audience? |
| Content conversion | Content → signup attribution | Is the content driving business outcomes? |
| Content efficiency | Cost per signup attributed to content | Is it cost-effective? |

**Content Program Health Score:**
```
Score = (Volume × 0.1) + (Quality × 0.3) + (Reach × 0.2) + (Conversion × 0.3) + (Efficiency × 0.1)
```

### 3.2 Community Program Evaluation

| Metric | How to Measure | Evaluation Criteria |
|--------|---------------|-------------------|
| Community growth | Net new members per month | Is the community growing? |
| Community health | DAM, answer rate, answer time | Is the community functioning? |
| Community value | Questions answered by community vs. staff | Is the community self-sustaining? |
| Community conversion | Community members who become paying customers | Is community driving revenue? |
| Community cost | Cost per active community member | Is it efficient? |

### 3.3 Event Program Evaluation

| Metric | How to Measure | Evaluation Criteria |
|--------|---------------|-------------------|
| Event reach | Registrations, attendees | Did we reach enough developers? |
| Event quality | NPS, satisfaction survey | Did developers find it valuable? |
| Event conversion | Attendees who sign up / activate within 30 days | Did the event drive adoption? |
| Event efficiency | Cost per activated developer from event | Is it cost-effective vs. other channels? |
| Speaker pipeline | New speakers identified and developed | Are we growing our speaker bench? |

### 3.4 Documentation Program Evaluation

| Metric | How to Measure | Evaluation Criteria |
|--------|---------------|-------------------|
| Docs coverage | % of API endpoints with examples | Is the documentation complete? |
| Docs quality | Time on page, scroll depth, search success rate | Is it effective? |
| Docs freshness | % of pages updated in last 6 months | Is it current? |
| Docs impact | TTFS, support ticket reduction | Does better docs improve outcomes? |
| Docs efficiency | Cost per documentation page maintained | Is it sustainable? |

## 4. Experiment Design

### 4.1 A/B Testing in DevRel

| Experiment | Hypothesis | Metric | Duration |
|-----------|-----------|--------|----------|
| Quickstart redesign | Simpler quickstart improves TTFS | TTFS, activation rate | 4 weeks |
| Interactive tutorial | Interactive beats text-only for completion | Tutorial completion rate | 4 weeks |
| Email sequence timing | Day-1 email improves activation vs. Day-3 | Day-7 activation rate | 8 weeks |
| Video quickstart | Video option improves activation for visual learners | TTFS by learning style | 4 weeks |
| Sandbox pre-population | Pre-populated data improves TTFS | TTFS, first API call rate | 4 weeks |

### 4.2 Experiment Framework

```
1. HYPOTHESIS: "If we [change], then [metric] will [improve/decrease] by [amount]"
2. DESIGN: A/B split with control group, statistical significance target (p < 0.05)
3. SAMPLE SIZE: Calculate required sample for statistical power (minimum 100 per group)
4. DURATION: Run for minimum 2 full cycles (if weekly pattern, minimum 2 weeks)
5. MEASURE: Track primary metric + guardrail metrics (nothing else gets worse)
6. ANALYZE: Compare treatment vs. control with confidence intervals
7. DECIDE: Ship winner, document learnings, plan next experiment
```

### 4.3 Quarterly Experiment Portfolio

| Quarter | Experiment | Expected Impact | Investment |
|---------|-----------|----------------|-----------|
| Q1 | Quickstart redesign | TTFS -30% | 2 weeks, 1 engineer |
| Q1 | Onboarding email A/B | Activation +5pp | 1 week, 1 advocate |
| Q2 | Interactive playground | Signup-to-call +10pp | 4 weeks, 1 engineer |
| Q2 | Video tutorial format | Tutorial completion +20% | 2 weeks, 1 advocate |
| Q3 | Community bot for FAQ | Answer time -50% | 3 weeks, 1 engineer |
| Q3 | Conference talk → blog pipeline | Content volume +25% | Ongoing, process change |

## 5. Program Lifecycle Decisions

### 5.1 Program Review Framework

Evaluate each program annually on four dimensions:

```
                  HIGH IMPACT
                      │
         GROW         │         MAINTAIN
    (increase         │      (keep current
     investment)      │       investment)
                      │
──────────────────────┼──────────────────────
                      │
         EXPERIMENT   │         CUT
    (try to improve,  │      (reduce or
     set deadline)    │       eliminate)
                      │
                  LOW IMPACT
                      │
    LOW EFFICIENCY         HIGH EFFICIENCY
```

### 5.2 Program Sunset Criteria

A program should be sunset (terminated) when:

1. It has been in the "Experiment" quadrant for 2+ quarters without improvement
2. The cost per outcome exceeds 3x the next best alternative
3. The target audience has shifted away from the channel
4. The program requires resources that would produce higher ROI elsewhere
5. Internal stakeholder support has evaporated (no one champions it)

**Sunset Process:**
1. Document the decision and rationale
2. Communicate to affected developers (do not just disappear)
3. Provide alternatives ("instead of X, use Y")
4. Archive materials for future reference
5. Redirect resources to higher-impact programs

## 6. Reporting and Communication

### 6.1 Stakeholder-Specific Reporting

| Audience | Cares About | Report Format |
|----------|-----------|---------------|
| CEO/CTO | Business impact, competitive position | Quarterly, 1-page executive summary |
| VP Engineering | Product quality, DX, developer satisfaction | Monthly, metrics dashboard |
| VP Marketing | Lead generation, brand awareness, content ROI | Monthly, funnel metrics |
| DevRel Team | Activity metrics, program health, experiments | Weekly, operational dashboard |
| Board | Revenue attribution, market position | Quarterly, 2-3 slides in board deck |

### 6.2 Narrative + Numbers

Every DevRel report should combine quantitative metrics with qualitative narrative:

```
METRIC: Activation rate increased from 28% to 33% this quarter (+5pp)

NARRATIVE: The quickstart redesign (launched Week 3) was the primary driver.
We observed a step-change in activation rate from 28% to 35% for developers
who onboarded after the redesign. The aggregate number (33%) is diluted by
developers who onboarded before the change. We expect the aggregate to reach
35% by next quarter as the pre-redesign cohort ages out.

NEXT STEP: We're now testing an interactive version of the quickstart that
we hypothesize will push activation above 40%.
```
